# STIP 开发计划

## 1. 项目目标

开发 **多源时空轨迹智能分析平台**，围绕人员、车辆、设备轨迹数据，构建从数据接入、实时位置、历史轨迹、空间查询、电子围栏、告警处理到统计分析的完整闭环。

项目采用分阶段交付，第一阶段优先做可演示 MVP，后续再扩展 Kafka、Flink、Spark、HDFS/HBase、Elasticsearch 等能力。

## 2. 当前状态

- 已有 `系统设计说明书.md`，包含完整项目背景、需求、架构、数据模型、接口和风险分析。
- 已创建项目级 `README.md`。
- 当前仓库尚未初始化后端、前端、数据库脚本和部署脚本。

## 3. 技术路线

第一阶段推荐技术栈：

- 后端：Java 17 + Spring Boot 3。
- 数据库：PostgreSQL + PostGIS。
- 缓存：Redis + Redis GEO。
- 前端：Vue 3 + TypeScript + Vite + Element Plus + ECharts + Leaflet。
- 部署：Docker Compose 启动基础依赖。
- 测试：JUnit 5、Testcontainers、Vitest。

后续阶段按需引入：

- Kafka：位置事件异步接入和消息削峰。
- Flink：实时清洗、去重、围栏判断、异常检测。
- Spark：离线统计、行为模式和历史趋势分析。
- HDFS/HBase：原始数据归档和海量轨迹存储。
- Elasticsearch：告警、日志和事件检索。

## 4. 阶段计划

### 阶段一：可演示 MVP

目标：先打通核心业务闭环，避免一开始被完整大数据架构拖慢。

任务：

- 初始化后端 Spring Boot 工程。
- 初始化前端 Vue 3 工程。
- 配置 PostgreSQL/PostGIS、Redis、Docker Compose。
- 设计并创建核心表：实体、轨迹点、电子围栏、告警事件。
- 实现位置事件上报接口。
- 实现实体最新位置查询。
- 实现历史轨迹查询和轨迹抽稀。
- 实现附近实体查询。
- 实现电子围栏基础管理。
- 实现异常速度或围栏进出基础告警。
- 实现前端地图监控、轨迹回放、围栏管理和告警列表。

验收标准：

- 本地可通过 Docker Compose 启动基础依赖。
- 后端接口可通过 OpenAPI 或接口测试工具访问。
- 前端可展示地图、实体位置、历史轨迹和告警列表。
- 核心逻辑具备单元测试或集成测试。
- README 包含真实启动、测试和构建命令。

### 阶段二：消息接入与实时计算

目标：体现高并发轨迹接入和实时处理能力。

任务：

- 引入 Kafka。
- 将位置接入改为写入 Kafka topic。
- 实现消费幂等、失败重试和异常消息记录。
- 引入 Flink 作业处理清洗、去重、异常速度、围栏判断。
- 将清洗结果写入 PostgreSQL/PostGIS、Redis 和告警 topic。

验收标准：

- 批量轨迹上报不直接阻塞数据库写入。
- Kafka 消息堆积、消费失败和重试行为可观察。
- Flink 作业可处理样例轨迹并输出清洗结果和告警。

### 阶段三：离线分析与大数据扩展

目标：补齐 Spark、HDFS/HBase 和历史分析能力。

任务：

- 归档原始轨迹数据到 HDFS。
- 使用 Spark 计算日统计、活跃实体、停留点 Top N 和移动距离。
- 评估 HBase 存储大规模轨迹明细或宽表的方案。
- 前端补充离线统计看板。

验收标准：

- 离线任务可基于样例数据重复执行。
- 统计结果可查询并在前端展示。
- 文档说明离线链路输入、输出和调度方式。

### 阶段四：工程化与展示强化

目标：形成稳定可演示、可说明、可复盘的项目。

任务：

- 完善 Docker Compose 一键启动。
- 完善 OpenAPI 文档。
- 补充数据库文档、接口文档、算法文档、部署文档和测试文档。
- 增加压测脚本和压测报告。
- 增加日志、指标和基础监控。

验收标准：

- 新环境可按文档启动项目。
- 测试、构建、部署步骤可复现。
- 压测结果能支撑核心接口性能说明。

## 5. 可复用性评估

- 标准位置事件模型可复用于人员、车辆、设备等实体。
- 距离计算、GeoHash、轨迹抽稀、停留点识别应封装为独立算法模块。
- 统一响应、异常处理、权限校验、分页模型应放入公共模块。
- 前端地图组件、轨迹播放组件、围栏绘制组件、告警表格组件应独立复用。
- Kafka 生产、消费、重试、死信处理应抽象为基础能力。

## 6. 可测试性评估

- 算法模块必须无外部强依赖，便于单元测试。
- PostGIS 空间查询使用 Testcontainers 做集成测试。
- Redis GEO 查询覆盖附近实体、实时位置缓存和过期策略。
- 接入接口覆盖单条上报、批量上报、参数错误和幂等场景。
- 前端覆盖地图筛选、轨迹回放、围栏表单和告警处理关键交互。
- 性能测试至少覆盖位置上报、历史轨迹查询和附近实体查询。

## 8. 风险提示

- 执行风险：完整实现所有微服务和大数据组件工作量较大，必须按阶段推进。
- 技术风险：PostGIS 空间索引、轨迹抽稀和历史查询性能需要尽早验证。
- 资源风险：本地同时运行大数据组件可能导致内存不足。
- 前端风险：大批量轨迹点直接渲染会卡顿，必须做抽稀和分页。
- 数据风险：真实人员或摄像头数据涉及隐私合规，开发阶段使用模拟数据。
- 错误风险：如果直接把所有轨迹点同步写数据库，容易出现写入瓶颈，阶段二需要 Kafka 削峰。
