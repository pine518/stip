# 多源时空轨迹智能分析平台系统设计说明书

## 1. 项目名称

**多源时空轨迹智能分析平台**

英文名称：`SpatioTemporal Insight Platform`

项目简称：`STIP`

## 2. 项目背景

在智慧城市、园区安防、车辆监管、物流调度、人员安全管理等场景中，业务系统会持续产生大量时空数据。这些数据来源复杂，既包括 GPS 定位、蓝牙定位、Wi-Fi 探针、摄像头抓拍识别结果，也包括传感器事件、业务日志和设备状态。

传统后台系统通常只解决业务数据的增删改查，难以支撑以下需求：

1. 高频位置数据实时接入。
2. 多源异构数据清洗、融合和标准化。
3. 海量轨迹数据的时空存储、空间检索和历史回放。
4. 基于轨迹的停留点识别、异常轨迹检测和行为模式分析。
5. 实时围栏告警、离线统计分析和可视化看板。
6. 高并发写入、大规模查询、消息削峰和缓存优化。

本项目以“人员、车辆、设备轨迹数据分析”为核心业务场景，设计一套覆盖后端架构、大数据处理、时空数据库、轨迹算法、微服务、API、消息队列、缓存和前端可视化的综合系统，用于满足 JD 中的主要技术栈和能力要求。

## 3. 项目可行性

### 3.1 业务可行性

轨迹分析是一个真实存在且需求稳定的业务方向，常见于：

1. 智慧园区人员定位与安全管理。
2. 城市车辆轨迹监管。
3. 物流运输路径追踪。
4. 安防系统人脸抓拍与轨迹串联。
5. 工业场景设备定位与传感器监测。
6. 外勤人员巡检轨迹管理。

这些场景天然需要多源数据融合、实时计算、空间查询、异常检测和可视化展示，能够完整覆盖 JD 中提到的技术能力。

### 3.2 技术可行性

项目可采用成熟技术组件实现：

1. 后端服务：Java 17 + Spring Boot 3 + Spring Cloud。
2. 消息队列：Kafka 承载高频轨迹事件。
3. 实时计算：Flink 实现清洗、去重、窗口聚合和告警。
4. 离线分析：Spark + HDFS/HBase 支撑批处理和历史分析。
5. 关系型与时空数据库：PostgreSQL + PostGIS 支撑轨迹点、围栏和空间查询。
6. 缓存：Redis GEO 缓存实时位置和热点查询。
7. 检索：Elasticsearch 支撑日志、轨迹事件和告警检索。
8. 前端：Vue3 + TypeScript + Element Plus + ECharts + Leaflet 实现管理后台和地图可视化。

### 3.3 开发可行性

建议分阶段实现，不一次性堆满全部大数据组件：

1. 第一阶段先实现单体增强版：Spring Boot + PostgreSQL/PostGIS + Redis + Vue3。
2. 第二阶段引入 Kafka，实现异步接入和消息削峰。
3. 第三阶段引入 Flink，实现实时清洗、停留点识别和告警。
4. 第四阶段引入 Spark/HDFS/HBase，实现离线统计分析。
5. 第五阶段补充压测、监控、文档和部署脚本。

该路线可以保证项目可落地，同时逐步覆盖 JD 技术栈。

## 9. 核心算法设计

### 9.1 轨迹清洗

处理规则：

1. 经纬度非法值直接丢弃。
2. 重复事件按 `event_id` 去重。
3. 乱序事件按事件时间窗口排序处理。
4. 速度超过业务阈值时标记为异常点。
5. 位置短时间大幅跳变且随后回归时标记为漂移点。

### 9.2 停留点识别

实现策略：

1. 按实体 ID 和时间排序轨迹点。
2. 使用滑动窗口累计连续点。
3. 当窗口内点位距离小于阈值且持续时间超过阈值，生成停留点。
4. 停留点坐标取窗口点位中心点或加权平均点。

参数示例：

| 参数 | 默认值 |
| --- | --- |
| 停留半径 | 100 米 |
| 停留时长 | 10 分钟 |
| 最少点数 | 3 |

### 9.3 异常轨迹检测

检测规则：

1. 速度异常：两点间距离 / 时间差超过阈值。
2. 轨迹漂移：短时间跳变距离过大，且后续点回到原区域。
3. 围栏异常：进入禁入区域或离开规定区域。
4. 离线异常：超过指定时间未上报位置。

### 9.4 路径还原

第一版建议采用简化策略：

1. 清洗异常点。
2. 按时间排序生成轨迹线。
3. 使用 Douglas-Peucker 算法进行轨迹抽稀。
4. 前端地图展示轨迹线、方向、速度和停留点。

后续可扩展道路匹配算法。

## 10. 接口设计

### 10.1 接口风格

1. 主要采用 RESTful API。
2. 接口统一使用 `/api/v1` 前缀。
3. 所有写接口支持 traceId。
4. 所有分页接口统一 `pageNo`、`pageSize`。
5. 所有接口返回统一响应结构。

### 10.2 核心接口

| 接口 | 方法 | 说明 |
| --- | --- | --- |
| `/api/v1/location-events` | POST | 上报单条位置事件 |
| `/api/v1/location-events/batch` | POST | 批量上报位置事件 |
| `/api/v1/entities/{id}/latest-location` | GET | 查询实体最新位置 |
| `/api/v1/entities/{id}/trajectory` | GET | 查询历史轨迹 |
| `/api/v1/geo/nearby` | GET | 查询附近实体 |
| `/api/v1/fences` | POST | 创建电子围栏 |
| `/api/v1/fences/{id}` | PUT | 更新电子围栏 |
| `/api/v1/alerts` | GET | 查询告警列表 |
| `/api/v1/alerts/{id}/handle` | POST | 处理告警 |
| `/api/v1/analytics/overview` | GET | 查询统计概览 |
| `/api/v1/analytics/stay-points` | GET | 查询停留点统计 |

### 10.3 统一响应格式

```json
{
  "code": "0",
  "message": "success",
  "traceId": "trace-20260625-000001",
  "data": {}
}
```

## 11. 部署设计

### 11.1 本地开发部署

本地开发建议使用 Docker Compose 启动基础依赖：

1. PostgreSQL + PostGIS。
2. Redis。
3. Kafka + Zookeeper 或 KRaft Kafka。
4. Elasticsearch。
5. Nacos。
6. MinIO 可选。

后端服务使用本地 Maven 启动，前端使用 npm 启动。

### 11.2 演示环境部署

演示环境建议使用 Docker Compose 部署：

```text
docker-compose.yml
  - postgres-postgis
  - redis
  - kafka
  - elasticsearch
  - nacos
  - gateway-service
  - auth-service
  - ingestion-service
  - trajectory-service
  - alert-service
  - analytics-service
  - admin-web
```

Flink 和 Spark 可在第一版中以本地任务或独立容器方式运行，避免部署复杂度过高。

### 11.3 生产部署设想

生产环境可扩展为 Kubernetes 部署：

1. 微服务以 Deployment 方式运行。
2. Gateway 通过 Ingress 暴露。
3. Kafka、PostgreSQL、Redis、Elasticsearch 使用独立集群或云托管服务。
4. Flink 使用 Flink Kubernetes Operator 管理任务。

## 15. 可复用性设计

1. 统一位置事件模型可复用于人员、车辆、设备等不同实体。
2. 轨迹清洗算法可独立封装为 `trajectory-algorithm` 模块。
3. GeoHash、距离计算、轨迹抽稀等工具可作为公共库。
4. Kafka 消息生产和消费封装为基础组件。
5. API 统一响应、异常处理、权限校验可复用于其他系统。
6. 前端地图组件、轨迹播放组件、告警列表组件可复用。

## 16. 可测试性设计

1. 算法模块无外部依赖，便于单元测试。
2. 数据库访问通过 Repository/Mapper 隔离，便于集成测试。
3. Kafka 消费逻辑支持本地嵌入式或 Testcontainers 测试。
4. 接口层通过 MockMvc 或 RestAssured 测试。
5. 前端核心组件可使用 Vitest 做组件测试。
6. 压测脚本纳入 `tests/performance` 目录，便于复现性能数据。

## 17. 风险提示

### 17.1 执行风险

1. 一次性实现全部微服务、大数据和算法模块，开发周期会明显拉长。
2. 本地机器如果内存不足，同时运行 Kafka、Flink、Spark、ES、HDFS、HBase 可能不稳定。
3. 地图能力可能依赖第三方地图服务，离线演示时需要准备本地瓦片或降级方案。
4. 摄像头抓拍和人脸识别不建议真实接入，可用模拟识别结果替代，避免隐私和合规问题。
5. HDFS/HBase/Spark 对面试展示有价值，但第一阶段不应作为核心阻塞项。

### 17.2 常见错误

1. 只做后台 CRUD，无法体现 JD 中的大数据和轨迹分析要求。
2. 所有数据直接写数据库，缺少 Kafka 削峰和异步处理设计。
3. 经纬度只存普通字段，不使用 PostGIS 空间索引。
4. 历史轨迹查询不分页、不抽稀，导致接口和前端性能问题。
5. 算法只写描述，没有样例数据和测试用例支撑。
6. 服务拆分过细但没有清晰边界，导致系统复杂度失控。

## 18. 推荐开发阶段

### 阶段一：可演示 MVP

目标：先实现核心闭环。

范围：

1. Spring Boot 单体或轻量多模块后端。
2. PostgreSQL/PostGIS。
3. Redis。
4. Vue3 管理后台。
5. 位置上报、实时位置、历史轨迹、附近查询、围栏管理。
6. 基础停留点识别和异常速度检测。

### 阶段二：消息与实时计算

目标：体现高并发和实时处理能力。

范围：

1. 引入 Kafka。
2. 位置接入改为异步写入。
3. 引入 Flink 清洗和告警。
4. 实现消费幂等、失败重试和死信策略。

### 阶段三：大数据离线分析

目标：覆盖 HDFS/HBase/Spark 能力。

范围：

1. 原始数据归档到 HDFS。
2. Spark 计算日统计和行为模式。
3. HBase 存储海量轨迹明细或宽表。
4. 前端展示离线分析结果。

### 阶段四：工程化与展示强化

目标：形成可面试展示项目。

范围：

1. Docker Compose 一键启动。
2. OpenAPI 文档。
3. 测试报告。
4. 压测报告。
5. 架构图、数据流图和算法说明。

## 20. 结论

推荐开发“多源时空轨迹智能分析平台”。该系统具备明确业务场景，可从 MVP 逐步演进到微服务和大数据架构，能够系统性覆盖 JD 中的高并发、分布式、微服务、数据库优化、时空数据、Kafka、Redis、Flink、Spark、HDFS、HBase、Elasticsearch、轨迹算法、RESTful API 和 Vue 前端能力。

该项目不建议一开始就完整实现所有组件。更合理的执行方式是先完成 `Spring Boot + PostgreSQL/PostGIS + Redis + Vue3` 的可演示闭环，再逐步接入 Kafka、Flink、Spark、HDFS/HBase 和监控压测能力。
